# EXTERNAL_FILES_ORCH.md

## Purpose

This file documents Markdown files outside the compiled source that the orchestration engine reads into prompts, recipe-compiler context, recipe execution, or orchestration UI context.

The important distinction:

- some files are **product contract files** that the engine itself expects to locate
- some files are **installed recipe/user input files** supplied by a running recipe
- some files are **generated runtime artifacts** written during orchestration execution

Existing authored JSON recipes can run without the recipe-compiler prompt files. For
example, `tag-team-run-plan.json` is already a valid recipe in the workflow/recipe
format, so normal recipe execution does not need `RECIPE_PROTOCOL.md` or
`orchestration\RECIPE_COMPILER_PROMPT.md`. Those two files matter when the engine is
compiling a user intent or natural-language request into a new recipe.
They are recipe authoring/compiler dependencies, not ordinary recipe execution
dependencies.

Throughout this document, `<install root>` means the engine's per-user install
directory under `%USERPROFILE%`.

## Product Contract Files

### `RECIPE_PROTOCOL.md`

Read by:

- `src/integration/orchestration-compiler.ps1`
- `Get-RecipeProtocolPath`
- `Get-CompilerPromptText`

Purpose:

- Canonical generated recipe protocol document for the workflow/recipe format.
- Appended into the AI recipe compiler prompt.
- Tells the compiler what schema, fields, adapters, routes, substitution tokens, and shapes are valid.
- Generated by `tools/dump-recipe-protocol.ps1`; should not be hand-edited.

Runtime behavior:

- `Get-CompilerPromptText` reads this file with `Read-TextFileSafe`.
- If this file cannot be found, compiler prompt construction returns empty and recipe compilation fails with `Compiler prompt documents were not found`.

Install status:

- `Inst.cmd` now copies this file to `<install root>\RECIPE_PROTOCOL.md`.
- `Install\makeInst.cmd` now includes this file in packaged installer inputs.

Risk:

- The installed engine can compile recipes only when `Find-RepositoryFile` can walk from the current roots back to the repo copy.
- That works when running from a repo context, but is not a proper installed-product dependency.

### `orchestration\RECIPE_COMPILER_PROMPT.md`

Read by:

- `src/integration/orchestration-compiler.ps1`
- `Get-CompilerPromptSourcePath`
- `Get-CompilerPromptText`

Purpose:

- Standing rules half of the AI recipe compiler prompt.
- Contains evergreen compiler rules: JSON only, required recipe-format fields, no unsupported concepts, use live adapter catalogs, etc.
- Does not contain adapter lists or field lists; those come from runtime and `RECIPE_PROTOCOL.md`.

Runtime behavior:

- `Get-CompilerPromptText` reads this file with `Read-TextFileSafe`.
- It is combined with:
  - compiler source
  - required schema from `Get-RecipeSchemaName`
  - live flow adapter list
  - live judgment adapter list
  - live action adapter list
  - user intent text
  - `RECIPE_PROTOCOL.md`

Install status:

- `Inst.cmd` now copies this file to `<install root>\orchestration\RECIPE_COMPILER_PROMPT.md`.
- `Install\makeInst.cmd` now includes this file in packaged installer inputs.

Risk:

- Same as `RECIPE_PROTOCOL.md`: installed recipe compilation depends on running from a location where repo files are discoverable.

## Orchestration Lab Context Files

### `src\docs\reference\ORCHESTRATION_MANUAL.md`

Read by:

- `src/ui/orchestration-lab.ps1`
- `Get-OrchestrationLabDefaultPlumbingDocPath`
- `Get-OrchestrationPromptText`

Purpose:

- Default Orchestration Lab "plumbing" context document.
- Included in the lab prompt when present.
- Gives the AI context about orchestration plumbing, hooks, queues, and recipe behavior.

Runtime behavior:

- Lab searches for the first existing path among:
  - `src\docs\reference\ORCHESTRATION_MANUAL.md`
  - `docs\reference\ORCHESTRATION_MANUAL.md`
  - `ORCHESTRATION_MANUAL.md`
  - `orchestration\HOOKS_PLAN_FINAL.md`
  - `HOOKS_PLAN_FINAL.md`
- If found, the selected document text is appended into the lab prompt.

Install status:

- **Handled by `Inst.cmd` if it lives under `src\docs`.**
- `Inst.cmd` copies `src\docs` to `<install root>\docs`.
- Current installed tree check: `<install root>\docs\reference\ORCHESTRATION_MANUAL.md` exists.

### `orchestration\HOOKS_PLAN_FINAL.md` / `HOOKS_PLAN_FINAL.md`

Read by:

- `src/ui/orchestration-lab.ps1`
- `Get-OrchestrationLabDefaultPlumbingDocPath`
- `Get-OrchestrationPromptText`

Purpose:

- Legacy/fallback plumbing context document for Orchestration Lab.
- Used only if higher-priority manual paths are absent.

Runtime behavior:

- Included as the "Plumbing Document" if it is the first existing fallback path.

Install status:

- **Not copied by normal install unless present under `src\docs` through another path.**
- Because `docs\reference\ORCHESTRATION_MANUAL.md` is installed, this fallback is usually not needed.

## Recipe-Provided External Markdown Inputs

These files are not hardcoded product dependencies. They are user- or recipe-supplied paths declared as artifacts in the workflow/recipe format. The engine reads them because the selected recipe says to read them.

### `src\recipes\tag-team-run-plan.json`

External markdown inputs:

- `$inputs.planPath`
- `$inputs.runPlanPath` default `RUN_PLAN.md`
- `$inputs.reviewGuidePath` default `PR_TOUGH.md`
- `$inputs.recipeRulesPath` default `RECIPE_RULES.md`
- `$inputs.constraintsPath`

Purpose:

- `planPath`: implementation plan being executed.
- `runPlanPath`: execution rules for implementer/fixer/closer workers.
- `reviewGuidePath`: review standard for reviewer worker.
- `recipeRulesPath`: recipe workflow rules.
- `constraintsPath`: per-plan constraints file.

Runtime behavior:

- Declared as `kind: "markdown"` artifacts.
- The recipe uses `core.assertFile`, prompt templates, `ai.run`, `judge.run`, and action steps to read and pass their content through orchestration prompts.

Install status:

- The recipe file itself is installed.
- The external markdown inputs are not installed because they belong to the target project or the operator's chosen plan.
- Current installed tree check: `<install root>\recipes\tag-team-run-plan.json` exists.

### `src\recipes\builtin.plan-executor.json`

External markdown inputs:

- `$inputs.planPath`
- `$inputs.runPlanPath` default `RUN_PLAN.md`
- `$inputs.reviewPromptPath` default `PR_TOUGH.md`

Purpose:

- Plan execution, implementation handoff, review handoff, and revision cycle.

Runtime behavior:

- Declared as `kind: "markdown"` artifacts.
- Used by recipe steps as source documents for worker prompts.

Install status:

- The recipe file is installed under `<install root>\recipes`.
- The external markdown inputs are supplied by the project/operator and are not copied by the engine's install.

### `src\recipes\make-a-plan.json`

External markdown output/input:

- `$inputs.planPath`

Purpose:

- Path where the original planning prompt markdown is written.
- The `make-plan` action then uses that path as the planning artifact root.

Runtime behavior:

- Declared as `kind: "markdown"` artifact `promptFile`.
- Written by `core.writeText`, then consumed by the `make-plan` action.

Install status:

- The recipe file is installed.
- The markdown file is user-selected output, not an install asset.

## Generated Runtime Markdown Artifacts

These are not external install files. They are generated per execution.

Examples:

- `$execution.tempDir/tag-team-implementation-prompt.md`
- `$execution.tempDir/tag-team-implementation-summary.md`
- `$execution.tempDir/tag-team-review-prompt.md`
- `$execution.tempDir/tag-team-plan-status.md`
- `$execution.tempDir/implementation-handoff.md`
- `$execution.tempDir/review.md`
- workflow run `prompt.md`
- workflow run `context.md`

Purpose:

- Preserve prompts, handoffs, summaries, and judge/review artifacts for one workflow execution.

Install status:

- Not installed.
- Created under orchestration runtime directories as workflows run.

## Conclusion

The orchestration runtime's recipe compiler has two product-owned external Markdown dependencies that install must ship:

1. `RECIPE_PROTOCOL.md`
2. `orchestration\RECIPE_COMPILER_PROMPT.md`

The installed-product fix is now in place:

- `RECIPE_PROTOCOL.md` is copied to `<install root>\RECIPE_PROTOCOL.md`
- `orchestration\RECIPE_COMPILER_PROMPT.md` is copied to `<install root>\orchestration\RECIPE_COMPILER_PROMPT.md`
- installer packaging inputs include both files
- `src\recipes` and `src\docs` behavior remains as-is
